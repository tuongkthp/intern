# OVS
### Provider-network
Đường đi từ vm ra internet ngoài:
    
    vm -> tap -> br-int -> patch -> br-ex -> port vật lý -> Hạ tầng mạng vật lý Router/Switch -> internet

Đường đi từ vm1 (Node1) sang vm2 (node2) cùng network:

    vm1 -> tap -> br-int -> patch -> br-ex -> port vật lý (node1) -> Switch -> port vật lý (node2) -> br-ex -> patch -> br-int -> tap -> vm2

Đường đi từ vm1 sang vm2 khác network:

    vm1 -> tap -> br-int -> patch -> br-ex -> port vật lý -> Switch vật lý -> router vật lý -> port vật lý -> br-ex -> patch -> br-int -> tap -> vm2

Quá trình xin cấp ip bằng DHCP của vm:

    vm -> DHCP discovery -> tap -> br-int -> br-tun -> node network/node controller -> br-tun -> br-int -> namespace -> dnsmasq reply -> Offer trở lại vm

### VPC
#### Đường đi từ vm ra internet ngoài:

    vm -> tap -> br-int -> patch -> br-tun -> encap (node compute) -> Hạ tầng mạng vật lý -> node (network/controller) -> br-tun -> patch -> br-int -> namespace router -> SNAT -> br-int -> patch -> br-ex -> port vật lý -> Hạ tầng mạng vật lý Router/Switch -> internet

Encap qua để đi qua tunnel: 

- options: local_ip=<IP của NIC tunnel> (Lựa chọn trong cấu hình)

- options: remote_ip=<IP node đích>

- UDP dst port

Khi không có floating ip, các vm nat chung 1 ip của router nên chỉ có thể từ vm kết nối ra ngoài chứ ko thể từ bên ngòai kết nối đến vm 

Khi có floating ip, vm được nat (SNAT/DNAT) riêng 1:1 cho interface nên có thể truy cập từ ngoài vào vm qua floating ip

#### Đường đi từ vm1 (Node1) sang vm2 (node2) cùng network:

    vm1 -> tap -> br-int -> patch -> br-tun -> encap (node compute) -> Hạ tầng mạng vật lý -> node 2 -> br-tun -> decap -> patch -> br-int -> tap -> vm2

#### Đường đi từ vm1 sang vm2 khác network:

    vm1 -> tap -> br-int -> patch -> br-tun -> encap (node compute) -> Hạ tầng mạng vật lý -> node (network/controller) -> br-tun -> decap -> patch -> br-int -> namespace router -> routing -> br-tun -> encap (network/controller) -> Hạ tầng mạng vật lý -> node compute -> br-tun -> decap -> patch -> br-int -> tap -> vm

Với DVR (Distributed virtual router) thì mỗi node compute sẽ có namespace router riêng nên l3 sẽ diễn ra nội bộ và nhanh hơn

# Quá trình tạo vm (OVS)

Phase 1–2: Auth & Scheduling — tương đối quen thuộc. nova-api validate token với Keystone, tạo DB entry, rồi nova-scheduler chạy filter chain (RAM, disk, AZ, compute capability) để chọn compute node.

Phase 3: Network Allocation — nova-compute gọi Neutron tạo port trước khi VM được spawn. Neutron ghi port vào DB với status=DOWN và binding:vif_type=ovs — tức là Neutron biết port này sẽ dùng OVS nhưng chưa có VIF thực tế nào trên hypervisor.

Phase 4: Storage — download image từ Glance về cache local, attach Cinder volume nếu có.

Phase 5: Spawn — libvirt tạo tap<id> (đây là interface VM thấy là eth0), tạo veth pair qvo/qvb, viết domain XML rồi gọi createWithFlags() để boot VM. Lúc này VM đã chạy nhưng mạng chưa thông.

Phase 6: OVS Wiring — đây là lúc neutron-ovs-agent làm việc. Agent polling ovsdb liên tục, phát hiện tap<id> mới xuất hiện, gọi neutron-server lấy port details rồi thực hiện hybrid plug: tạo Linux Bridge qbr, nối tap<id> + qvb vào qbr, nạp iptables Security Group rules, cắm qvo vào br-int với local VLAN tag. Sau đó báo neutron-server port ACTIVE, neutron-server gửi event network-vif-plugged về nova-compute, instance mới được đánh dấu ACTIVE.

Điểm quan trọng: VM đã boot từ Phase 5 nhưng DHCP chỉ có thể thành công sau khi Phase 6 hoàn tất — vì đường đi tap → qbr → br-int → br-tun → VXLAN → controller → qdhcp phải được wiring đầy đủ thì DHCP Offer mới đến được VM.
